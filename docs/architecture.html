⚡ 快速回答(本文档你最关心的两点)

沙箱由哪个模块创建? 由 Backend 创建。具体是 Backend 里的 SandboxManager(编排:查 sandbox_registry 复用健康沙箱,否则新建)+ DockerProvisioner(实际 docker run 起容器,DooD 借宿主 Docker daemon)。在回合开始时由 turn 端点调用 manager.get_endpoint_for_user(user_id) 触发。Worker 和 Sandbox 自己都不创建沙箱。

Backend 和 Worker 都能任意水平扩展吗? Worker 可以(完全无状态,任一实例服务任一回合;只差 Backend 当前指向单一 worker_endpoint,前置一个负载均衡即可)。Backend 不能"任意"扩:加实例能提升吞吐、且 DB 会话锁保证跨实例不会重复跑同一会话(正确性 OK);但可重连回合的状态(TurnHub/ActiveTurn)在进程内存里,所以"续看/取消"必须路由到正在跑该回合的同一实例(按会话 sticky 路由),或引入 post-v1 的 pub/sub 来共享回合事件。详见 §5 水平扩展分析。

1全景架构

自上而下四层,三条跳。绿 = 无状态、可弃可换;橙 = 有状态、需持久。安全边界:Worker(脑)持 LLM Key、可信;Sandbox(手)无 Key、能力最小化、不可信。

① 前端 / 客户端
聊天 UI(块流渲染) + 文件管理 · localStorage 记住 user/session
│   SSE + REST(公网)  发消息/取消/文件=REST,回合输出=SSE
② Backend(FastAPI · 有状态)
鉴权(待建)/ 会话与消息 / 配置文档 / 文件编排 / 会话锁 / 压缩编排 / 沙箱编排 / 进程内 TurnHub(可重连回合) / 沙箱 reaper
│   gRPC 流式(内网)  RunTurn / RunTurnStream / Summarize
③ Worker「脑」(无状态 · 持 LLM Key · 可信)
agent-loop(LLM→工具→回填) · provider 抽象(OpenAI 兼容) · 拼 system 提示词 · 每次调用独立
│   gRPC ExecTool(内网)  worker 按需连该用户沙箱执行工具
④ Sandbox「手」(无状态计算 · 无 Key · 不可信)
bash / read_file / write_file · cap_drop=ALL · no-new-privileges · mem/cpu/pids 限额
│
⑤ 持久层
Postgres(用户/配置/会话/消息/记忆/技能/沙箱登记)· 每用户 workspace 卷(挂进沙箱 /workspace,跨会话/跨 agent 共享)· 对象存储(上传/快照,规划中)
无状态(可水平扩展/随意替换) 有状态(持久 / 需协调)

核心洞察:稳定性来自"状态放在哪",而非"计算活多久"。把对话/配置/文件/记忆搬到持久层后,Worker、Sandbox 这两层计算单元即可随意替换、横向扩容、互不拖累。

2组件职责与状态性

前端 客户端态

  • React19 + Vite + TS + Tailwind(浅色 + teal)
  • SSE 消费回合流;块流时序渲染(思考/正文/工具)
  • 断网/切会话用 GET 续看;停止按钮取消
  • 状态仅本地(localStorage 当前 user/session),非权威

Backend 混合

  • FastAPI;唯一访问 Postgres 的层
  • 会话锁(DB,租约式)保证会话内串行 + 跨实例不重复跑
  • 编排:组装请求 / 调 Worker / 落库 / 压缩 / 沙箱路由
  • 进程内 TurnHub/ActiveTurn:可重连回合的事件缓冲(非共享)
  • 后台:沙箱 reaper 循环 + 关停时 drain_hub 释放锁

Worker「脑」 无状态

  • gRPC 服务;Servicer 仅持一个 provider_factory
  • 持 LLM Key(可信);每次 RunTurn/Summarize 完全独立
  • 跑 agent-loop;按需连 request.sandbox_endpoint 执行工具
  • 回合间不保留任何 per-user/per-session 状态

Sandbox「手」 计算无状态

  • gRPC 服务;只暴露 ExecTool(bash/读写文件)
  • 无 LLM Key;cap_drop=ALL + no-new-privileges
  • 挂载该用户 workspace 卷到 /workspace(状态在卷上)
  • 每用户一个容器,空闲 TTL 回收

Postgres 有状态

  • 系统的状态锚:user / agent_config / session / message
  • context_document / memory_entry / skill / sandbox_registry
  • 会话锁、压缩摘要、沙箱登记都在这里协调

每用户 workspace 卷 有状态

  • 宿主 <host_root>/<user_id>/workspace
  • 跨会话、跨 agent、跨沙箱重建稳定共享
  • pip --user / npm -g 依赖持久于此

3一次回合(流式)的完整流程

以前端发一条消息为例(POST /sessions/{id}/turn/stream):

  1. 加锁:Backend 在 DB try_acquire 会话锁(租约式);并发同会话直接 409。
  2. 落 user 消息:写入 messages 并 commit。
  3. 拿沙箱:调 SandboxManager.get_endpoint_for_user(user_id) —— 查 sandbox_registry,健康则复用,否则 DockerProvisioner.spawn 起一个并登记(见 §4)。
  4. 组装请求:build_run_turn_request 拼出 system(环境+配置文档+记忆+技能+历史摘要)+ 过滤后的历史 + 本条消息 + 沙箱端点。
  5. 起独立 Runner:asyncio.create_task 跑回合(断连不取消);注册到进程内 TurnHub;同时返回 SSE 订阅流。
  6. BE→Worker 流式:Backend 经 gRPC RunTurnStream 把请求发给 Worker。
  7. Worker agent-loop:调 LLM → 若有工具调用,经 gRPC ExecTool 连该用户沙箱执行 → 回填 → 再调 LLM,直到收尾。增量(思考/正文/工具)实时流回。
  8. 缓冲 + 落库:Backend 把事件写进 ActiveTurn(供断线补播)并转 SSE;TurnDone 时把助手/工具消息落库(仅成功时落,无半成品)。
  9. 回合后压缩:按真实 context_tokens 超阈值则折叠历史进 session.summary(仍持锁、心跳续租)。
  10. 透明自愈:中途超窗→压缩重试、瞬时错误→退避重试,重试前发 reset 让前端清屏;耗尽才给可恢复/不可恢复提示。
  11. 收尾:释放会话锁;从 Hub 移除。断线的客户端可用 GET /turn/stream 续看(补播+实时)。

4沙箱的创建与生命周期

谁创建:Backend。 编排在 backend/sandbox/manager.py 的 SandboxManager,实际起容器在 backend/sandbox/docker_provisioner.py 的 DockerProvisioner(DooD:借宿主 Docker daemon)。Worker / Sandbox 自身都不创建沙箱。

阶段发生了什么
触发回合开始,turn 端点调 get_endpoint_for_user(user_id)。懒创建:没有回合就没有沙箱。
复用查 sandbox_registry 是否有该用户的 active 沙箱;有且探活通过 → touch 续期并直接复用(一个用户一个沙箱,跨会话/跨 agent 共享)。
探活/自愈探活失败(沙箱崩溃/backend 重启后端点失效)→ 标记 dead、尽力停掉、落到下方重建。
新建DockerProvisioner.spawn:docker run 一个容器,挂 <host_root>/<user_id>/workspace → /workspace,带 cap_drop=ALL、no-new-privileges、mem/cpu/pids 限额、/tmp tmpfs。在事务外起(可能慢)。
登记/防竞态写 sandbox_registry;partial unique index 处理并发抢建——输的一方停掉自己那个、路由到赢家,避免一个用户被劈成两个沙箱。
隔离容器只挂该用户目录,宿主其它用户目录在容器内不存在 → 跨用户越权读被堵。
回收Backend 后台 _reaper_loop 周期跑 reap_idle:空闲超 TTL 的沙箱先停再标 dead;跳过仍有 running 会话的用户,避免长回合被中途回收。
provisioner 可换: 配置 sandbox_provisioner=docker(真隔离,dev/prod)或 inprocess(进程内,CI/无 Docker)。两者实现同一 SandboxProvisioner Protocol,对上层透明。

5水平扩展分析

这是你问题的详细回答。结论:Worker 可任意扩;Backend 可扩吞吐但"续看/取消"需配合 sticky 路由;Sandbox 按用户绑卷;Postgres 是状态锚走常规 DB 扩展。

组件有状态?水平扩展?关键约束 / 怎么扩
Worker「脑」 无状态(仅持 Key 配置) ✅ 可任意 任一 worker 可服务任一回合;每次调用独立、按需连沙箱。
唯一现状限制:Backend 配置指向单一 worker_endpoint。多 worker 时在其前面放一个负载均衡 / 服务发现即可(或上 post-v1 的消息队列派发)。架构上无障碍。
Backend 混合:权威状态在 Postgres,但可重连回合的 Hub 在进程内 ⚠️ 吞吐可扩,"任意"需配合 加实例即提升并发吞吐;DB 会话锁(租约)保证跨实例不会重复跑同一会话(正确性已安全)。
约束:回合作为独立 asyncio 任务跑在"收到 POST 的那个实例",其 SSE 事件缓冲在该实例的进程内 TurnHub。于是 GET /turn/stream(续看)和 POST /turn/cancel 必须打到同一实例,否则别的实例 Hub 里没有这个回合 → 续看得到 204、取消变 no-op。解法:按会话 sticky 路由(把同一 session 的请求固定到同一实例),或引入 post-v1 的 pub/sub 共享回合事件(设计 §14)。沙箱 reaper / SandboxManager 是每实例各跑,登记在共享 DB、幂等且有防竞态,多实例并存安全。
Sandbox「手」 计算无状态,但挂着该用户持久卷 ⚠️ 加主机扩,单用户绑卷 每用户一个容器,懒建、空闲回收;按需加沙箱主机扩容。
但某用户的沙箱固定在其卷所在主机,不能随意迁移(用户卷多区域 / 跨 AZ 是 post-v1)。
Postgres 有状态(系统状态锚) 常规 DB 扩展 读副本 / 连接池 / 按需分片。它是"无状态计算"成立的前提,扩展方式与一般有状态 DB 相同。
一句话: 把 Backend 做到"任意实例接任意请求"还差一步——把进程内 TurnHub 换成共享事件总线(pub/sub),这正是设计 §14 的 post-v1 项「第②跳消息队列 + pub/sub」。在那之前,多实例 Backend 用会话级 sticky 路由即可正确运行(发起、续看、取消都落到同一实例),吞吐随实例数线性增长。

6安全分层

脑 / 手 分层

  • Worker 持 LLM Key、可信;Sandbox 无 Key、不可信
  • 沙箱即便被工具产出的恶意代码攻陷,也拿不到 Key

沙箱能力最小化

  • cap_drop=ALL + no-new-privileges(故 apt/sudo 失败,符合预期)
  • mem/cpu/pids 限额;/tmp 走限额 tmpfs

租户卷隔离

  • 容器只挂该用户目录;跨用户目录在容器内不存在
  • 文件 API 有路径越狱(path jail)防护

出网

  • 出网默认放行(装包/抓取);allow_net=False 故意 fail-loud(应在部署网络层用 allowlist/代理收紧)
  • 本机 gRPC 绕过代理(no_grpc_proxy)
当前缺口: 还没有鉴权/多租户授权(前端 localStorage 假 user、后端信任传入 user_id)。这是上线前必补的第一块,见 roadmap.html · P1。
