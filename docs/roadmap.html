0愿景与现状一句话

愿景(源自设计 §1):把"一个用户焊死一个进程"的脆弱架构,拆成有状态后端 + 无状态 agent 服务—— 状态(对话/配置/文件/记忆)放在持久层,计算单元(worker/sandbox)随时可替换、可横向扩容、互不拖累。 稳定性来自"状态放在哪",而非"计算活多久"。

现状一句话:核心闭环已经跑通且可靠——用户能在隔离沙箱里和 agent 多会话对话、用工具读写持久文件、 断网刷新可续看、上下文满了自动压缩、回合失败会透明自愈。 但它目前更像一个「单人可用的强原型」:还没有真正的鉴权与多租户、没有智能体记忆、 没有成本/用量可观测、没有规模化派发。Roadmap 即围绕把它变成"可对外托管的云"。

前端 / 客户端(聊天 UI + 文件管理) │ ① SSE + REST(公网) 后端 Backend(有状态:Postgres / 配置 / 会话 / 文件编排 / 编排锁 / 压缩) │ ② gRPC 流式 Worker「脑」(无状态:agent-loop · provider · 持 LLM Key · 可信) │ ③ gRPC ExecTool Sandbox「手」(无状态:bash/读写文件 · cap_drop · 无 Key · 不可信) │ 每用户持久 workspace 卷(跨会话/跨 agent 共享)

2关键缺口(Roadmap 的依据)

把"强原型"变成"云"还差四类东西。前两类(鉴权、记忆)是设计里已点名或低估的,后两类(可观测、规模)是产品/运营必需:

缺口现状为什么要紧
鉴权 / 多租户零鉴权;前端 localStorage 假 user;后端信任任意传入的 user_id无法对外、无法隔离用户、无法计费——是"云"的前提
智能体记忆仅数据管道骨架(读取+追加 + 注入 prompt);无写入工具/抽取/检索/UI/遗忘"跨会话记住偏好与事实"是 agent 产品的核心价值,目前几乎 inert
可观测 / 成本回合 usage 已采集,但无面板、无配额、无追踪LLM 产品烧钱;不可见 = 不可控、不可运营、不可信任
弹性 / 规模BE↔Worker 直连;无队列派发、无池化/自动扩缩设计目标正是"横向扩容、互不拖累";直连下扛不住多用户

3推荐推进顺序

两条路线取决于近期目标。下面是我的建议序列(可按你的优先级调整):

  1. 先打地基(P1) —— 认证 → 授权 → 密钥/限流。没有它,任何对外动作都不安全;且它是计费/配额/市场的前置。
  2. 补最大能力空白(P2 记忆) —— 自动提炼写入 + pgvector 检索,让 agent 真正跨会话变聪明。可与 P1 部分并行(记忆不依赖鉴权)。
  3. 装上仪表盘(P3 成本/可观测) —— 一旦多用户在用,成本与排障立刻成为刚需;与 P1 配额天然互补。
  4. 按需扩容(P4) —— 当并发/负载成为瓶颈时再上队列派发与池化;过早做是过度工程。
  5. 深化体验(P5) —— 核心稳固后,按产品差异化需要选做 steering / 分支 / 原生端。
决策点:如果近期目标是"尽快给真实用户用" → 严格 P1 优先;如果是"先把 agent 做强、再考虑对外" → 可把 P2(记忆 + 工具)提到 P1 之前,鉴权延后。两者都成立,取决于你要先要"可托管"还是先要"更聪明"。

4贯穿性原则(每个阶段都遵守)

  • brainstorm → spec → plan → TDD 实现 → 对抗审查:沿用至今的工作流,每个功能先出设计/计划文档再动手。
  • 无状态边界不破:新能力优先放进后端(有状态)或保持 worker/sandbox 无状态、可替换;DB 仍后端独占。
  • 尽力而为 + 可恢复:失败要么透明自愈(已做),要么给用户清晰、可执行的提示;绝不静默坏掉成功的回合。
  • 安全分层:Key 只在 worker;沙箱不可信、能力最小化;新工具/出网默认收紧。
  • 测试是基线:后端 testcontainer 集成 + worker 单测 + 前端 vitest;回归全绿才提交。
